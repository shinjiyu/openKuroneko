# 钉钉消息接入调研报告

> 调研时间：2026-03-08  
> 调研目标：钉钉开放平台的接入流程、权限要求，以及单聊、群聊、图片、语音消息的API实现方式

---

## 一、接入流程

### 1.1 基本接入步骤

钉钉开放平台的应用接入流程包括以下关键步骤：

1. **创建开发者账号**  
   访问 [钉钉开发者后台](https://open.dingtalk.com)，注册并登录开发者账号。

2. **创建应用**
   - 选择应用类型：企业内部应用 或 第三方企业应用
   - 填写应用基本信息（名称、描述、图标等）
   - 获取 AppKey 和 AppSecret（用于 API 调用鉴权）

3. **配置应用能力**
   - 添加「机器人」能力（用于消息收发）
   - 配置消息接收模式：推荐使用 **Stream 模式**
   - 设置消息接收地址（Webhook URL）

4. **申请权限**
   - 在「权限管理」页面申请所需接口权限
   - 部分权限需要企业管理员审核通过

5. **发布应用**
   - 配置完成后发布应用
   - 应用发布后可在企业内使用

### 1.2 API 调用流程

```
获取 Access Token → 调用业务 API → 处理响应结果
```

**获取 Access Token：**
- 接口：`POST https://oapi.dingtalk.com/gettoken`
- 参数：`appkey` 和 `appsecret`
- 返回：`access_token`（有效期 2 小时，需缓存）

## 二、权限要求

### 2.1 权限机制

钉钉开放平台采用 **Scope 权限机制**：
- 权限以应用为维度授予
- 每个应用的权限独立存在
- 多个应用调用同一接口需分别申请权限

### 2.2 权限类型

分为两类：
1. **委托权限（Delegated Permission）**：代表已登录用户访问资源
2. **应用权限（Application Permission）**：以应用身份访问资源

### 2.3 消息相关权限

发送消息所需的主要权限：

| 权限点 | 说明 |
|--------|------|
| `im:message` | 消息基础权限 |
| `im:message:send_as_bot` | 以机器人身份发送消息 |
| `im:resource` | 媒体资源权限（图片、文件等） |
| `ChatBot.Read` | 机器人读取权限 |
| `ChatBot.Write` | 机器人写入权限 |
| `Card.Instance.Write` | 创建和投放卡片实例 |
| `Card.Streaming.Write` | 卡片流式更新 |

## 三、消息 API 实现方式

### 3.1 单聊消息

#### 3.1.1 机器人发送单聊消息

**方式一：服务端 API**

- 接口：`POST https://api.dingtalk.com/v1.0/robot/oToMessages/batchSend`
- 需要权限：`im:message:send_as_bot`
- 请求示例：
```json
{
  "robotCode": "dingxxxxxx",
  "userIds": ["user123"],
  "msgKey": "sampleText",
  "msgParam": "{\"content\":\"你好，这是测试消息\"}"
}
```

**方式二：工作通知消息**

- 接口：`POST https://oapi.dingtalk.com/topapi/message/corpsend`
- 特点：企业向员工发送工作通知，显示在「工作」频道

#### 3.1.2 普通消息（员工个人发送）

需要前端配合：
1. 调用 JSAPI 唤起联系人选择页面
2. 获取会话 `cid`
3. 调用服务端接口发送消息

---

### 3.2 群聊消息

#### 3.2.1 机器人发送群聊消息

**方式一：服务端 API**

- 接口：`POST https://api.dingtalk.com/v1.0/robot/groupMessages/send`
- 请求示例：
```json
{
  "robotCode": "dingxxxxxx",
  "openConversationId": "cidxxxxxx",
  "msgKey": "sampleText",
  "msgParam": "{\"content\":\"群消息内容\"}"
}
```

**方式二：Webhook（自定义机器人）**

- 获取群机器人的 Webhook URL
- 发送 POST 请求：
```bash
curl -X POST \
  'https://oapi.dingtalk.com/robot/send?access_token=xxxxx' \
  -H 'Content-Type: application/json' \
  -d '{"msgtype":"text","text":{"content":"群消息内容"}}'
```

#### 3.2.2 创建群聊会话

- 接口：`POST https://api.dingtalk.com/v1.0/im/chat/groups`
- 需要权限：`Chat.Group.Create`

### 3.3 图片消息

#### 3.3.1 实现流程

图片消息需要两步操作：

**步骤 1：上传媒体文件获取 media_id**

- 接口：`POST https://oapi.dingtalk.com/media/upload`
- 参数：
  - `access_token`：访问令牌
  - `type`：文件类型（`image`）
  - `media`：图片文件（multipart/form-data）
- 返回：
```json
{
  "errcode": 0,
  "media_id": "xxx",
  "created_at": "1234567890"
}
```

**media_id 特点：**
- 可复用（同一个 media_id 可多次使用）
- 仅能在钉钉客户端内使用
- 不占用组织钉盘空间

**步骤 2：发送图片消息**

```json
{
  "msgKey": "sampleImageMsg",
  "msgParam": "{\"photoURL\":\"@lALPDfJ6V_FPDpDNAfzNAfQ\"}"
}
```

#### 3.3.2 图片格式要求

- 支持格式：jpg、png、gif、bmp
- 文件大小限制：通常不超过 2MB

---

### 3.4 语音消息

#### 3.4.1 客户端录音

使用钉钉 JSAPI：
- `dd.device.audio.startRecord` - 开始录音
- `dd.device.audio.stopRecord` - 停止录音
- `dd.device.audio.play` - 播放音频
- `dd.device.audio.translateVoice` - 语音转文字

#### 3.4.2 发送语音消息

**流程：**
1. 上传音频文件获取 `media_id`
2. 调用消息发送接口

**音频格式要求：**
- 推荐格式：AMR、AAC
- 文件大小：通常不超过 2MB
- 时长限制：根据具体接口而定

#### 3.4.3 API 调用示例

```json
{
  "msgKey": "sampleAudio",
  "msgParam": "{\"media_id\":\"xxx\",\"duration\":5}"
}
```

## 四、消息类型汇总

钉钉机器人支持的消息类型：

| 消息类型 | msgKey | 说明 |
|----------|--------|------|
| 文本消息 | `sampleText` | 纯文本内容 |
| 图片消息 | `sampleImageMsg` | 图片（需先上传获取 media_id） |
| 语音消息 | `sampleAudio` | 音频文件 |
| 文件消息 | `sampleFile` | 文件（需先上传） |
| 链接消息 | `sampleLink` | 图文链接 |
| Markdown | `sampleMarkdown` | Markdown 格式 |
| ActionCard | `sampleActionCard` | 交互卡片 |
| OA 消息 | `sampleOa` | OA 消息格式 |

| 对比项 | 钉钉 | 飞书 |

## 六、开发建议

### 6.1 推荐架构

```
┌─────────────┐
│  钉钉客户端  │
└──────┬──────┘
       │ Stream / Webhook
       ▼
┌─────────────┐
│   消息网关   │ ← 处理消息收发
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  业务逻辑层  │ ← 消息处理、AI 对接
└─────────────┘
```

### 6.2 关键注意事项

1. **Access Token 缓存**：避免频繁调用，建议缓存并设置刷新机制
2. **权限申请**：提前规划所需权限，避免开发中途因权限不足受阻
3. **错误处理**：妥善处理 API 返回的错误码，实现重试机制
4. **安全性**：AppSecret 保密，验证消息签名防止伪造

### 6.3 参考资源

- 钉钉开放平台：https://open.dingtalk.com
- API 文档：https://open.dingtalk.com/document/development/api-overview
- 开发者百科：https://open-dingtalk.github.io/developerpedia

## 七、总结

钉钉开放平台提供了完整的消息 API 能力：

1. **接入流程清晰**：创建应用 → 配置权限 → 发布使用
2. **权限管理严格**：采用 Scope 机制，部分权限需管理员审核
3. **消息类型丰富**：支持文本、图片、语音、文件、Markdown、卡片等多种类型
4. **媒体文件处理**：需先上传获取 media_id，再发送消息（与飞书类似）
5. **多种接入方式**：支持服务端 API、Webhook、JSAPI 等

整体接入难度适中，建议优先使用企业内部应用 + Stream 模式实现机器人消息收发。
