## 四、安全边界总结

### 4.1 有效的安全边界

1. **文件系统沙箱**:
   - read_file 工具限制在工作目录和临时目录
   - 无法通过 read_file 访问父目录

2. **进程隔离**:
   - Agent 进程独立运行
   - 资源使用受 ulimit 限制

3. **框架保护**:
   - .brain/ 目录由框架自动管理
   - 探索操作不会破坏框架文件

4. **网络边界**:
   - 可发起 HTTP 请求
   - 无法监听新端口（未测试，推测受限）

### 4.2 边界不一致发现

**约束声明**: "禁止访问父目录或同级目录"  
**实际情况**: 
- ❌ read_file("../package.json") → Security violation
- ✅ shell_exec("cat ../package.json") → 成功

**结论**: 约束仅限制 read_file 工具，Shell 命令不受此限

**风险评估**: 低风险（只读访问，无写权限）

### 6.2 改进建议

#### 对约束库的建议
```markdown
[避坑] 文件操作工具（read_file）仅限工作目录和临时目录
— 但 Shell 命令（shell_exec）不受此限制，可访问父目录内容（只读）
```

#### 对框架的建议
1. **统一沙箱边界**: 让 Shell 命令也遵守文件系统沙箱
2. **磁盘空间监控**: 添加磁盘空间告警机制
3. **临时文件清理**: 自动清理超过 N 天的临时文件

#### 对 Agent 的建议
1. 定期执行 `rm /tmp/openkuroneko/*/business-*.tar.gz` 清理导出文件
2. 大文件操作后立即清理
3. 避免在磁盘空间紧张时执行导出操作
