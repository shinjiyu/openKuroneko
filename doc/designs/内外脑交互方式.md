# 内外脑交互方式

外脑与内脑之间**仅通过文件 + 进程**交互，无 RPC/HTTP。每个内脑实例有独立的 `workDir` 和 `tempDir`，一一对应。

---

## 一、目录约定

| 目录 | 路径 | 谁创建 | 用途 |
|------|------|--------|------|
| **workDir** | `<obDir>/tasks/<instanceId>/` | 外脑池子 launch 时 | 内脑 cwd；产出报告、.brain/ 等 |
| **tempDir** | `<OPENKURONEKO_TMP>/<agentId>/` | 外脑池子 + 内脑 identity | 与 workDir 一一对应（`deriveAgentId(workDir)`）；放 input/output/status/directives 等 |

内脑进程以 `--dir <workDir>` 启动，内部用同一算法算出 tempDir，故与外脑共用同一 tempDir。

---

## 二、外脑 → 内脑（下发）

### 1. input 文件（`<tempDir>/input`）

| 项目 | 说明 |
|------|------|
| **谁写** | 外脑：`set_goal` 时池子写入；也可作为通用「外脑指令」通道（若未来扩展） |
| **谁读** | 内脑：BLOCKED 态每 tick 读（`safeReadInput`）；DECOMPOSE/EXECUTE/SLEEPING 也会检查是否有新 input（外脑干预） |
| **格式** | 纯文本。新任务：`[NEW_GOAL]\norigin_user: xxx\norigin_thread: xxx\n\n<目标正文>`；其它为自由文本指令 |
| **消费** | 内脑通过 `input.offset` 增量消费，读后不删文件，只推进 offset |

用途：下发新目标、外脑干预（打断/重规划）、BLOCKED 时等用户回复前也可由外脑写指令。

### 2. directives 文件（`<tempDir>/directives`）

| 项目 | 说明 |
|------|------|
| **谁写** | 外脑：`send_directive` 工具追加一行 JSON；PushLoop 在 BLOCK 解封时追加 `[BLOCK解封] 用户回复：xxx`（type: feedback） |
| **谁读** | 内脑：BLOCKED 态读并清空；EXECUTE 前也会读并清空（约束/需求注入 constraints.md） |
| **格式** | JSON Lines：`{"ts":"...","type":"constraint"|"requirement"|"feedback","content":"...","from":"user_id"}` |
| **消费** | 读后**清空文件**（readAndClearDirectives） |

用途：BLOCK 解封、补充约束/需求、给内脑的即时指令。

### 3. goal.md（`<workDir>/.brain/goal.md`）

| 项目 | 说明 |
|------|------|
| **谁写** | 外脑池子 launch 时写入（与 input 内容一致）；内脑 BLOCKED 处理 [NEW_GOAL] 时也会写 |
| **谁读** | 内脑：DECOMPOSE 读目标、readGoalOriginUser/readGoalOriginThread 读头部元数据 |

用途：保证首 tick 就能读到目标，避免「空 goal → BLOCK」的竞态。

### 4. 进程生命周期

| 操作 | 说明 |
|------|------|
| **启动** | 外脑池子 `spawn(launchCommandTemplate, { cwd: workDir })`，内脑以子进程运行 |
| **停止** | 外脑 `stop_inner_brain` → 池子对实例发 SIGTERM，等待退出后标记 EXITED |

---

## 三、内脑 → 外脑（上报）

### 1. output 文件（`<tempDir>/output`）

| 项目 | 说明 |
|------|------|
| **谁写** | 内脑：controller 通过 ioRegistry.output.write() **追加** JSON 行（BLOCK/COMPLETE/PROGRESS） |
| **谁读** | 外脑 PushLoop：按 **offset** 增量读（`output.ob.offset`），解析行后分发 |
| **格式** | 每行一个 JSON：`{"type":"BLOCK"|"COMPLETE"|"PROGRESS","message":"...","target_user":"?", "question":"?", "ts":"..."}` |
| **消费** | 外脑读后只推进 offset，不删文件 |

用途：BLOCK → 外脑发「任务阻塞通知」并等用户回复后写 directives 解封；COMPLETE → 外脑发完成通知；PROGRESS → 仅日志（可选推送）。

### 2. status 文件（`<tempDir>/status`）

| 项目 | 说明 |
|------|------|
| **谁写** | 内脑：controller 每 tick 开始时 `syncStatus()` 写一整份 JSON |
| **谁读** | 外脑：`getInnerStatus()`（对话/参与决策用）、`read_inner_status` 工具、PushLoop 的 getGoalOriginUser/getGoalOriginThread |
| **格式** | 见下方「status 字段说明」 |

用途：外脑只读内脑当前「状态机阶段、里程碑、是否阻塞、任务发起人」等，不写。

#### status 字段说明（详细）

| 字段 | 类型 | 含义 |
|------|------|------|
| **ts** | string (ISO 8601) | 本状态快照的写入时间，每次 syncStatus() 更新 |
| **mode** | string | 内脑状态机当前阶段。取值：`DECOMPOSE`（战术拆解）、`EXECUTE`（执行里程碑）、`ATTRIBUTE`（归因）、`BLOCKED`（等待外脑/人工）、`SLEEPING`（睡眠/循环间歇） |
| **milestone** | object \| null | 当前**活跃**里程碑（仅 EXECUTE 时有意义）。`id`（如 M1）、`title`（标题）、`cyclic`（是否循环里程碑） |
| **goal_origin_user** | string \| null | 当前任务发起人的 user_id，来自 goal.md 头部 `origin_user: xxx`。用于 BLOCK/COMPLETE 通知路由 |
| **blocked** | boolean | 是否处于阻塞态。**派生字段**：恒等于 `(mode === 'BLOCKED')`，与 mode 一致 |
| **block_reason** | string \| null | 当 mode 为 BLOCKED 时的原因，如 `goal.md 缺失`、`目标已完成，等待新目标`、Decomposer 失败原因等 |
| **sleeping_until** | string \| null | 仅当 **mode === 'SLEEPING'** 时存在。计划唤醒时间（ISO 8601） |
| **cycle_count** | number | 仅当 **mode === 'SLEEPING'** 时存在。当前循环里程碑已完成的轮次 |

说明：外脑类型 `InnerBrainStatus` 中还有 `block_question`，该字段**不写入 status 文件**，仅出现在 output 的 BLOCK 事件里。

## 四、交互一览表

| 方向 | 载体 | 路径 | 触发/时机 |
|------|------|------|-----------|
| 外→内 | input | tempDir/input | set_goal 写入；内脑 BLOCKED/DECOM/EXEC/SLEEP 读 |
| 外→内 | directives | tempDir/directives | send_directive、BLOCK 解封 写入；内脑 BLOCKED/EXEC 前读并清空 |
| 外→内 | goal.md | workDir/.brain/goal.md | 池子 launch 写；内脑 DECOMPOSE 读 |
| 外→内 | 进程 | spawn/kill | 池子启动/停止内脑进程 |
| 内→外 | output | tempDir/output | 内脑 BLOCK/COMPLETE/PROGRESS 追加；PushLoop 轮询按 offset 读 |
| 内→外 | status | tempDir/status | 内脑每 tick syncStatus 写；外脑 getInnerStatus/read_inner_status/ PushLoop 读 |

---

## 五、外脑工具与交互的对应关系

| 工具 | 交互方式 |
|------|----------|
| **set_goal** | 池子创建 workDir+tempDir，写 input + goal.md，写 status 初值，spawn 内脑 |
| **send_directive** | 向指定实例的 tempDir/directives 追加一行 JSON |
| **read_inner_status** | 读该实例 tempDir/status（及可选的 .brain/milestones.md） |
| **list_inner_brains** | 池子列表 + 各实例 tempDir/status、workDir/.brain/milestones.md |
| **stop_inner_brain** | 池子对实例进程发 SIGTERM |

外脑对话时用的 `getInnerStatus()` 直接读「当前任意一例运行中实例」的 tempDir/status，用于拼系统提示里的内脑状态摘要。

## 六、外脑如何获取内脑的详细展示信息（里程碑与日志）

内外脑之间**没有**额外 RPC，所有「详细展示用」的信息都来自**外脑主动读内脑占用的目录**。池子持有每个实例的 `workDir` 与 `tempDir`，外脑（或挂在外脑上的 WebChat 服务）用这些路径去读文件即可。

### 6.1 数据来源一览

| 展示内容 | 外脑读取路径 | 谁写 | 说明 |
|----------|--------------|------|------|
| **当前状态**（mode、blocked、当前里程碑标题） | `<tempDir>/status` | 内脑每 tick syncStatus | JSON，见第四节 status 字段 |
| **完整里程碑列表**（M1/M2…、状态、标题、描述） | `<workDir>/.brain/milestones.md` | 内脑 Decomposer / 控制器 | 外脑解析文本行为结构化列表 |
| **内脑运行日志**（逐条 JSONL） | `<tempDir>/logs/YYYY-MM-DD.jsonl` | 内脑 Logger 模块 | 外脑按需 tail 最近 N 行 |

也就是说：**里程碑**来自 workDir 的 `.brain` 产出，**日志**来自 tempDir 的 `logs`，**状态**来自 tempDir 的 `status`。外脑只读不写。

### 6.2 WebChat 的实现方式

WebChat 适配器（`src/channels/adapters/webchat.ts`）在提供 HTTP API 时，通过 **pool** 拿到每个实例的 `workDir`、`tempDir`，然后读文件并返回：

| API | 行为 |
|-----|------|
| **GET /webchat/inner-status** | 读**任一运行中实例**的 `tempDir/status`，返回 JSON（兼容旧单实例时可读 `innerStatusFile`）。 |
| **GET /webchat/inner-brain/list** | 对池中**每个**实例：读 `tempDir/status` → 拼成 `status`；读 `workDir/.brain/milestones.md` → 用 `parseMilestones()` 解析成 `milestones` 数组；检查 `tempDir/logs` 是否有 `.jsonl` → `hasLogs`。返回列表，每条含 id、poolStatus、originUser、goal 摘要、startedAt、exitedAt、**status**、**milestones**、hasLogs。 |
| **GET /webchat/inner-brain/logs?lines=N&instanceId=xxx** | 根据 `instanceId` 找到该实例的 `tempDir`（未传则用最新一例），读 `tempDir/logs/` 下当日 `.jsonl` 文件，取最后 N 行解析为 JSON 数组返回。 |

前端（`webchat-ui.html`）：

- **顶部状态栏**：每 5 秒调一次 `/webchat/inner-brain/list`，取第一个实例的 `status`，显示 mode、是否阻塞、当前里程碑标题。
- **侧边栏「内脑」**：同样用 list 渲染多个实例入口；点击某一实例打开**内脑监控面板**（`#ib-panel`）。
- **内脑监控面板**：每 3 秒同时请求 `list` 与 `logs?lines=500&instanceId=当前实例`，用 `renderIbStatus(inst)`、`renderIbMilestones(inst.milestones)`、`renderIbLogs(data.logs)` 刷新**状态、完整里程碑列表、日志列表**。

因此，**你之前做 WebChat 时能看到的「详细里程碑和日志」**，就是通过上述两个 API 从外脑读内脑的 `workDir/.brain/milestones.md` 和 `tempDir/logs/*.jsonl` 实现的；入口在侧边栏点击对应内脑实例，打开监控面板即可看到。

### 6.3 与旧版 chat-ui（单实例）的对比

| 方式 | 数据从哪来 | 如何到前端 |
|------|------------|------------|
| **chat-ui/**（`chat-ui/server.mjs`） | 单实例：固定一个 `BRAIN_DIR`、`OUTPUT_FILE`、`TEMP_DIR` | 服务端轮询：每 500ms 读 output 并 broadcast；每 300ms tail 日志并 broadcast；每 2s 读全部 `.brain` 文件（含 milestones.md）并 broadcast。前端 SSE 收 `output` / `log` / `brain` 事件。 |
| **WebChat 适配器**（多实例） | 池子中每个实例的 workDir + tempDir | 前端轮询：调 `/webchat/inner-brain/list` 和 `/webchat/inner-brain/logs`，服务端按需读对应实例的 milestones.md 与 logs。 |

共同点：都是**外脑（或独立服务）读内脑写好的文件**，没有内脑主动推送；差异在于单实例时用 SSE 推「整份 brain + 日志流」，多实例时用 HTTP 按实例拉 list + 按需拉 logs。

## 七、内脑产物回传（文字 + 文件/图片）

内脑完成时可能需回传：**一段话**、**单个/多个文件**、**图片**，或组合。约定见 [inner-brain-deliverables](../protocols/inner-brain-deliverables.md)。

- **内脑侧**：Executor 通过工具 **register_deliverable(relative_path)** 将「相对于 workDir 的路径」登记到 `<tempDir>/deliverables.json`。Controller 在全部里程碑完成、写入 COMPLETE 前读取该文件，将路径列表写入 COMPLETE 事件的 `deliverables` 字段，读后删除该文件。
- **外脑侧**：PushLoop 解析 COMPLETE 时，若存在 `deliverables` 则仅将这些路径与 workDir 拼接后作为附件来源；否则回退为扫描 workDir。附件按类型（image/file 等）与大小/数量上限构造，通过 ChannelRegistry.send 下发；各频道适配器支持**多条附件**（先发文本，再按条发送每个附件）。
