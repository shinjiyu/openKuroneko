# 飞书实现对比与优化方案

对比 **openKuroneko 当前飞书适配器** 与 **OpenClaw 飞书扩展** 的差异，并给出分阶段优化方案。  
OpenClaw 分析详见：`doc/designs/openclaw-飞书消息处理分析.md`。

---

## 1. 能力对比总表

| 维度 | openKuroneko 现状 | OpenClaw | 差距说明 |
|------|-------------------|----------|----------|
| **接入模式** | Webhook + WebSocket（@larksuiteoapi/node-sdk） | Webhook + WebSocket | 基本一致 |
| **身份主键** | union_id 优先，回退 open_id；DM thread 用 canonical | open_id 为主，user_id 回退 | 我们偏 union_id，OpenClaw 偏 open_id；发 DM 我们需 getOpenIdForUnionId |
| **发送方展示名** | 无 | contact/v3/users 解析 + 10min 缓存，可关 | 我们未做，Agent 只能看到 ID |
| **@ 收：是否 @ 了 bot** | union_id + open_id 双匹配，支持 app_id | open_id + mentions + post 内 mentionedOpenIds + @_all | 我们缺 post 内 @、@_all |
| **@ 收：正文处理** | `content.replace(/@\S+/g, '').trim()` 简单去 @ | normalizeMentions：key→`<at user_id="…">名</at>`，仅去掉 bot 自己的 @ | 我们丢失“谁被 @”的结构，且可能误删正常文本 |
| **@ 发：回复时 @ 别人** | 无 | mentionTargets，回复首条带 @（文本/卡片两种格式） | 我们无法“回复并 @ 某人” |
| **引用：入站** | reply_to = parent_id | parent_id + 拉取被引用消息正文 → quotedContent，拼进 body | 我们未拉取引用内容，Agent 不知道“回复的是哪句话” |
| **引用：出站** | 有 reply_to 时调 reply 接口 | reply 接口 + 撤回 230011 时 fallback 发新消息 | 我们未做撤回 fallback |
| **引用：话题** | 未用 root_id/thread_id | root_id/thread_id 用于话题会话与 reply_in_thread | 我们事件类型未声明 root_id/thread_id |
| **发文本** | 仅 msg_type: text | 富文本用 post（md 结构），表格等用卡片 | 我们无富文本/卡片，长文无排版 |
| **发图片/文件** | 支持，reply 时也带 payload | 同，且 reply_in_thread 可选 | 我们发回复时未传 reply_in_thread |
| **图片下载** | im/v1/images（image_key） | messageResource.get（message_id + file_key, type: image） | 我们方式对“消息里的图”可能不适用部分场景（OpenClaw 文档称 im/v1/images 仅限 im 上传的图） |
| **文件下载** | 未实现（附件仅保留 feishu-file://） | messageResource.get type: file，落盘 + 占位符 | 我们未下载文件给 LLM |
| **post 内嵌图** | post 只提纯文本，未抽 imageKeys | parsePostContent 抽 imageKeys，逐个 messageResource 下载 | 我们 post 里的图未给 Agent |
| **多账号** | 无 | 多 account，listAccountIds / resolveAccount | 我们单应用单实例即可，可不做 |
| **群会话 scope** | 无 | group / group_sender / group_topic / group_topic_sender | 我们未做话题级会话 |
| **去重** | event_id（webhook）+ message_id（ws），内存 TTL | 内存 + 持久化去重，防重连重放 | 我们无持久化去重 |
| **日志** | logger.info 若干事件 | 结构化 + 事件名 + 摘要，符合 M11 规范 | 我们可对齐项目 logging 规范 |

---

## 2. 风险与体验缺口（简要）

- **@ 正文**：用 `@\S+` 抹掉整段会误伤正常内容（如邮箱、@ 符号）；且下游无法知道“被 @ 的人”，无法做 @ 转发。
- **引用**：不拉取被引用内容，Agent 不知道用户在回复哪句，容易答非所问。
- **回复接口**：被回复消息撤回时我们直接报错，OpenClaw 会 fallback 发新消息。
- **图片**：消息附件/ post 内嵌图若走 im/v1/images 可能部分场景失败，OpenClaw 统一用 messageResource。
- **发消息**：仅文本、无 post/卡片，长文与代码块体验差；且回复时未带 reply_in_thread，话题内回复可能“飘”到群主时间线。

---

## 3. 优化方案（分阶段）

### 3.1 原则

- **P0**：修错与明显体验问题（@ 解析、引用内容、回复 fallback、图片下载 API）。
- **P1**：体验增强（发 post/卡片、回复时 @ 别人、reply_in_thread、发送方展示名）。
- **P2**：可选增强（话题会话、持久去重、文件下载落盘、多账号、日志规范对齐）。

不追求与 OpenClaw 完全一致，优先闭环“能正确理解、正确回复、不报错”。

---

### 阶段一（P0）：正确性与稳定性

| 序号 | 项 | 做法 | 产出 |
|------|----|------|------|
| P0-1 | **@ 正文规范化** | 参考 OpenClaw：用 `event.message.mentions` 的 key→open_id/name 做替换；仅去掉“机器人自己”的 @，其余替换为 `<at user_id="open_id">显示名</at>` 或保留占位；不再用 `@\S+` 整段删除。 | 入站 content 保留“谁被 @”且不误伤正文；为后续 @ 转发打基础。 |
| P0-2 | **引用内容拉取** | 入站若存在 `message.parent_id`，调飞书「获取消息」接口取被引用消息正文；拼成 `[Replying to: "…"]\n\n` 前缀写入 content，或写入 InboundMessage 新字段 `quoted_content`，由下游拼进 prompt。 | Agent 能看见用户“回复的是哪句”，减少答非所问。 |
| P0-3 | **回复接口 Fallback** | 调用 `messages/:id/reply` 时若返回 230011（或 231003 等“已撤回/不存在”），自动改为调用「发送新消息」接口发到同一会话，不向用户抛错。 | 被引用消息被撤回时仍能正常回复。 |
| P0-4 | **图片下载 API 对齐** | 消息内图片（含 image 类型、post 内嵌）统一改为使用 `im/v1/message_resource/get`（message_id + file_key/image_key，type=image）；若无则保留现有 im/v1/images 作回退。 需确认当前飞书文档对“消息中的图”的推荐 API。 | 避免部分场景图片下载失败或接口废弃。 |
| P0-5 | **事件类型补全** | 在 FeishuEventBody 的 message 中增加 `root_id?`、`thread_id?`，解析后写入 InboundMessage（如 `reply_root_id?`、`thread_id?`），供后续话题与 reply_in_thread 使用。 | 为 P1 的 reply_in_thread 与 P2 话题会话提供数据。 |

**交付标准**：用户“引用某条消息”提问时，Agent 能拿到被引用内容并正确回复；回复接口在被撤回时自动降级为新消息；@ 不再误删正文；图片下载方式与飞书推荐一致。

---

### 阶段二（P1）：体验增强

| 序号 | 项 | 做法 | 产出 |
|------|----|------|------|
| P1-1 | **发送方展示名** | 入站时用 open_id（或 union_id）调 contact/v3/users 解析展示名，缓存 10 分钟；在 InboundMessage 或给 Agent 的上下文中增加 `sender_name`；可配置开关以省 API 配额。 | 群聊/历史中显示“谁在说话”，便于 Agent 与用户理解。 |
| P1-2 | **回复时带 reply_in_thread** | 出站若存在 `msg.reply_to` 且会话为群聊，在调用 reply 接口时增加 `reply_in_thread: true`（若飞书 API 支持）；若无 root_id 则可不带。 | 回复落在话题线内，不“飘”到主时间线。 |
| P1-3 | **发 post 或卡片** | 发文本时：若内容含 Markdown（如代码块、表格），可改为飞书 post 结构或交互卡片（如 openclaw buildMarkdownCard）；否则保持 text。需评估飞书 post 与卡片的 API 与限制。 | 长文/代码块排版更好。 |
| P1-4 | **@ 转发（回复并 @）** | 入站若“同时 @ 了 bot 和他人”，解析出 mentionTargets（排除 bot）；出站调用 reply_to_user 时支持传入 `mention_user_ids`（或等价的 open_id 列表），飞书适配器在首条回复中拼上 `<at user_id="…">名</at>`（文本或卡片格式）。 | 满足“请回复并 @ 张三”的场景。 |
| P1-5 | **post 内 @ 与 @_all** | 收消息时若 message_type 为 post，从 content 解析出 mentionedOpenIds（与 OpenClaw parsePostContent 一致）；且若原始 content 含 `@_all` 视为 @ 了 bot。 | 群内富文本 @ 与 @所有人 判定正确。 |

**交付标准**：群聊中能看到发言人名字；回复引用时尽量进话题线；支持“回复并 @”；富文本 @ 与 @所有人 正确；重要回复可带简单卡片/ post。

---

### 阶段三（P2）：可选增强

| 序号 | 项 | 做法 | 产出 |
|------|----|------|------|
| P2-1 | **话题级会话** | 群聊且存在 root_id/thread_id 时，thread_id 或会话 key 包含 topic 维度（如 feishu:group:chatId:topic:rootId），历史与路由按话题聚合。 | 多话题群内会话不串线。 |
| P2-2 | **持久化去重** | 对 message_id（及可选 event_id）做持久化去重（如 SQLite/文件），重启或重连后仍能过滤重放。 | 避免重连导致重复处理。 |
| P2-3 | **文件下载与占位** | 对 file/audio 类附件用 messageResource.get 下载并落盘，在给 Agent 的 content 中插入占位符（如 `<media:file:文件名>`），并传递路径或 URL 供工具使用。 | LLM 可基于文件内容回答。 |
| P2-4 | **文件名编码** | 上传文件时对非 ASCII 文件名做 RFC 5987 编码（与 OpenClaw sanitizeFileNameForUpload 一致），避免 form-data 导致静默失败。 | 中文等文件名上传稳定。 |
| P2-5 | **日志规范** | 飞书相关日志统一走项目 Logger 模块，事件名与字段对齐 M11（如 input.read、send、tool 等），便于复盘与排查。 | 符合项目规范、可复盘。 |

---

## 4. 实施建议

1. **先做 P0**：P0-1～P0-5 可并行或按表顺序做；P0-2（引用内容）和 P0-3（fallback）对“引用”体验影响最大。
2. **协议与配置**：若新增 InboundMessage 字段（如 `quoted_content`、`reply_root_id`、`sender_name`）或 OutboundMessage 的 `mention_user_ids`，在 `doc/protocols/` 或类型定义处补一笔，便于其它频道与下游对齐。
3. **测试**：每项尽量有“飞书真实事件”或 mock 用例：尤其是 @（含 post）、引用、回复接口 fallback、图片下载。
4. **文档**：每阶段完成后更新 `doc/designs/openclaw-飞书消息处理分析.md` 中“openKuroneko 实现要点”一节，保持与 OpenClaw 的对比可追溯。

---

## 5. 小结

| 阶段 | 目标 |
|------|------|
| **P0** | 引用可读、回复不崩、@ 不误伤、图片接口正确、事件字段完整 |
| **P1** | 谁在说话、回复进话题、可回复并 @、富文本 @、简单富文本发送 |
| **P2** | 话题会话、持久去重、文件落盘、文件名编码、日志规范 |

按 P0 → P1 → P2 推进，可在不重写的前提下，逐步对齐 OpenClaw 的飞书体验并规避当前主要风险。
