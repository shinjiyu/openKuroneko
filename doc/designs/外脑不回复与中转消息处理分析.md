# 外脑不回复与中转消息处理分析

基于 2026-03-09 日志与代码路径梳理。

---

## 零、外脑会「一直尝试发私聊」吗？外脑有心跳吗？

### 外脑没有发给用户的「心跳」

- 外脑**没有**定时向用户发心跳或保活消息。
- 唯一的定时器是 **PushLoop**：每 **2 秒**轮询一次所有内脑实例的 **output 文件**，解析 `[BLOCK]` / `[COMPLETE]` / 其他行（PROGRESS）。这是「读内脑输出」的轮询，不是「给用户发消息」的定时任务。

### 什么会触发外脑主动发私信？

只有两类逻辑会**主动**往用户发消息（包括私信）：

| 触发源 | 机制 | 目标 thread |
|--------|------|-------------|
| **1. 内脑 COMPLETE** | PushLoop 读到内脑写的 `[COMPLETE]` → `handleComplete` → 向 `goal_origin_user` 发「任务完成」通知 | `getBestThreadId`：优先 **UserStore 里该用户 priority 最高的 channel**，通常是 **DM**（如 `feishu:dm:xxx`），没有则 fallback 到 ThreadStore 里该用户的 DM，再没有则用 `origin_thread`（群） |
| **2. 内脑 BLOCK** | PushLoop 读到 `[BLOCK]` → `handleBlock` → **BlockEscalationManager.waitForResolution** → 按该用户的 channel 列表（**先 DM 后群**）依次发「任务阻塞通知」，等用户回复解封 | 先试 `feishu:dm:xxx`（或 dingtalk:dm:xxx），失败或超时（默认 30min）再试下一个 channel，最后可通知 owner_users |

因此，如果你感觉「一直收到私信」或「一直尝试发私信」，通常是：

- **多次 COMPLETE**：多个任务完成或内脑多次写出 COMPLETE，每次都会向目标用户发一条完成通知，且优先发到 DM。
- **多次 BLOCK 升级**：内脑多次 BLOCK（例如 goal 为空、多个实例同时 BLOCK），每次都会走一遍 escalation，先试 DM 发阻塞通知；若发送失败（如日志里 dingtalk 404），会打 `escalation.send_fail`，30 分钟后会升级到下一 channel 或 owner。

### 日志里能看到什么

- `push-loop`：`inner_output`（BLOCK/COMPLETE）、`complete.notify`（发给了谁、thread_id）、`block.start_escalation`。
- `block-escalation`：`escalation.start`、`escalation.sent`（成功）、`escalation.send_fail`（失败）、`escalation.owner_notified`（通知了 owner）。
- 飞书私信回复：`msg.received`（你先发）→ `conversation.start` reason=dm → `loop.send`/`loop.reply`，这是**被动回复**，不是外脑主动心跳。

结论：**外脑没有心跳；主动发私信只由「任务完成通知」和「任务阻塞通知」触发，且都优先选 DM。** 若想减少私信，可考虑让完成/阻塞通知优先发群（需改 UserStore 的 channel priority 或 getBestThreadId 的优先级逻辑）。

## 一、为什么很多提问消息外脑都不回复？

从日志里可以看到，未回复的群消息主要被标成两类：**mention_others** 和 **silent**。

### 1. `group.skip` reason = **mention_others**（占多数）

**逻辑（见 `outer-brain/index.ts`）：**

- 群消息且 **没有 @ 本 bot**（`is_mention === false`）
- 但 **存在 mentions**（`msg.mentions?.length > 0`，即飞书事件里带了“被 @ 的人”）
- → 外脑认为：**用户 @ 了别人，不是 @ 我** → 只记入 thread 历史，不回复。

**日志典型例子：**

- 用户发「@_user_1 你好」→ `mention_open_ids: ["ou_1ef8dff044cb9fd3552b0ec40614bb4d"]`（被 @ 的是别人）→ `is_mention: false` → **mention_others**，不回复。
- 用户发「@_user_1 我是谁」「@_user_1 再看我是谁」「@_user_1 哥哥」「@_user_1 我看看你work不 哥哥」「@_user_1 你看到几个角色」「@_user_1 那不就是一共有三个人么」等，飞书里 mentions 都是 **on_94626acdda9676a3ff1aee6c9c58bf73**（另一个用户/另一个 bot），本实例的 `bot_union_id` 是 **on_efa363b520e0656c14a823b813e38094**，对不上 → `is_mention: false` → **mention_others**，不回复。

**原因归纳：**

- 飞书事件里的 `mentions` 表示的是 **被 @ 的账号列表**（open_id/union_id）。
- 群里有多个 Kuroneko 或多个“机器人”时，用户点选 @ 的可能是 **另一个实例/另一个 bot**，则 mentions 里没有本实例的 union_id/open_id。
- 外脑严格按「mentions 里是否包含本 bot」判断是否 @ 我；不包含就一律视为「用户在和别人说话」→ 不回复。

若要本实例回复，用户需要 **明确 @ 到本实例**（在飞书里选到的是本 bot），使事件里出现本实例的 union_id/open_id，才会 `is_mention: true` → `conversation.start` → 回复。

### 2. `group.skip` reason = **silent**

**逻辑：**

- 群消息、**没有 @**（`is_mention === false`）、**没有 mentions**（或前面已排除 mention_others）
- 进入「非 @ 群消息参与决策」：调用 `participationEngine.shouldSpeak(msg, recentText, soul, innerStatus)`。
- 若返回 `false` → 只记入 thread，打 `group.skip` reason=**silent**，不回复。

**shouldSpeak 会因以下原因返回 false（见 `participation.ts`）：**

- `proactive_level === 0`：完全沉默，只回复 @。
- 冷却期内、或 5 分钟内主动发言次数已达上限。
- `level === 1` 时：只回答「直接问句」（句尾 `?` / `？`），否则不回复。
- 消息太短（trim 后 `< 3` 字符）：视为表情/语气词，不调 LLM。
- 快速模型判断「不该说话」。

**日志例子：**

- 「你们俩先商量你们演什么」「怎么都不说话了」「我让你们说话」「你们俩先互相认识一下」→ 无 @、无 mentions，走 shouldSpeak → 被判 **silent**，不回复。

## 二、中转广播 vs 对话消息：处理路径差异（设计如此）

### 对话消息（飞书事件 → InboundMessage）

1. 飞书推送事件 → Feishu 适配器 `parseEvent()` → 得到 `InboundMessage`（或 null，如自身消息已过滤）。
2. 适配器调用 `onMessage(msg)` → 外脑 **onInboundMessage(msg)**。
3. 外脑按规则分支：
   - 私信 → 直接 **runConversation(msg)**，回复。
   - 群 + **is_mention** → **runConversation(msg)**，回复。
   - 群 + 有 mentions 但非 @ 我 → **mention_others**，只 **appendUser**，不回复。
   - 群 + 其他 bot 发言（sender_type === 'app'）→ **other_bot**，只 appendUser，不回复。
   - 群 + 无 @ → **shouldSpeak** → true 则 runConversation，false 则 **silent**，只 appendUser。

所以：**对话消息**会触发「是否回复」的完整决策（mention / mention_others / proactive / silent），并可能进入 `runConversation` 生成回复。

### 中转广播（relay type=broadcast）

1.  Feishu 适配器在 WebSocket 里收到 relay 的 **type=broadcast**（其它 agent 的发言）。
2. 只执行：**relayIngestRef.current(threadId, userId, content, ts)**（见 `feishu.ts`）。
3. 外脑侧该回调实现为（见 `outer-brain.ts`）：
    - `threadStore.ensureThread(threadId, 'feishu', ts)`
    - `threadStore.appendUser(threadId, userId, content, ts)`
4. **不调用 onInboundMessage**，也**不调用 runConversation**。

因此：

- **中转广播只做“写进当前 thread 历史”**，等价于把「其它 agent 的这条发言」当成一条 user 消息插入上下文。
- **不会**对这条广播触发回复、不会走 SPEAK/SILENT 决策。
- 之后若**用户再发一条消息**（走正常 InboundMessage 路径），外脑在构建上下文时会看到这条广播内容，再决定是否回复用户的新消息。

这样设计的目的：避免多个 agent 对「别人刚说的那一句」抢着回复，只让各 agent 在**用户新消息**或**明确 @ 自己**时再参与。

## 三、总结

| 现象 | 原因 |
|------|------|
| 很多群消息不回复 | 多为 **mention_others**：飞书 mentions 里是被 @ 的人，若用户 @ 的是另一个 Kuroneko/别人，本实例认为没 @ 自己 → 不回复；少部分为 **silent**：未 @ 且 shouldSpeak 为 false。 |
| 中转来的消息“和对话消息处理不一样” | **设计如此**：中转广播只 **appendUser** 进 thread，不经过 onInboundMessage，不触发回复；只有后续用户新消息才会再走完整决策并可能回复。 |

若希望本实例在群里更多回复，可考虑：

1. **@ 行为**：让用户明确 @ 到本实例（保证 mentions 里出现本实例 union_id）。
2. **参与策略**：在 soul 里调高 `participation.proactive_level`、放宽冷却与 5min 上限，或调整 shouldSpeak 的 prompt/规则（仍要注意避免刷屏）。
