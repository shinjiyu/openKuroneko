# 消息中转服务 — 设计讨论

**目的**：飞书不把「其它机器人发言」推给我们，导致群聊记录断档。通过**一个中心化中转服务器**，让各 agent 在群里的发言被广播到其它 agent，由各自的**飞书插件侧**把收到的内容插进本机的群聊记录；**agent 本体完全不了解这些细节**。

---

## 1. 架构概览

```
                    ┌─────────────────────────────────────┐
                    │           中转服务器 (Relay)          │
                    │  - 注册表：agent_id ↔ 回调/订阅信息    │
                    │  - 收到「发言」→ 向其它已注册方广播    │
                    │  - 单一 key 鉴权                      │
                    └─────────────────────────────────────┘
                      ▲ 注册/建长连接            ▲ 经长连接广播
                      │                         │
        ┌─────────────┼─────────────┐   ┌───────┴───────┐
        │             │             │   │  长连接推送到  │
        │             │             │   │  各 agent 的   │
        │             │             │   │  飞书插件      │
        ▼             ▼             ▼   ▼               ▼
   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
   │ Agent A │  │ Agent B │  │ Agent C │  │ Agent B  │  │ Agent C  │
   │ 飞书插件 │  │ 飞书插件 │  │ 飞书插件 │  │ 飞书插件 │  │ 飞书插件 │
   └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘
        │            │            │            │            │
        │ 发群消息后   │            │            │ 收到广播 →  │ 收到广播 →
        │ 上报 Relay  │            │            │ 插入本机    │ 插入本机
        │            │            │            │ 群聊记录    │ 群聊记录
        ▼            ▼            ▼            ▼            ▼
   ┌─────────┐  ┌─────────┐  ┌─────────┐
   │ Agent A │  │ Agent B │  │ Agent C │   ← agent 核心不感知
   │ 外脑/本体│  │ 外脑/本体│  │ 外脑/本体│     注册与插入仅在
   └─────────┘  └─────────┘  └─────────┘     飞书插件侧实现
```

- **中转服务器**：独立部署；维护「谁注册了」；收到某 agent 的「我在某群发了言」后，向**其它**已注册的 agent 做广播（不发给发言人自己）。
- **飞书插件侧**（每个 agent 的 Feishu 适配层）：
  - **注册**：启动时向中转服务器注册（如 agent_id、可被调用的 URL 或 WebSocket）。
  - **上报**：本 agent 在飞书群里发完消息后，把「thread_id / 发言者 id / content / ts」发给中转服务器。
  - **接收广播**：被中转服务器调用时，把收到的「某群、某 agent 的发言」**插入到本机该群的 thread 历史**（只写存储，不触发回复）。
- **Agent 本体**：不参与注册、上报、插入；完全不知道中转的存在。

---

## 2. 参与方与职责

| 角色 | 职责 |
|------|------|
| **中转服务器** | 提供注册接口、接收发言并广播；做简单鉴权（见下）。 |
| **飞书插件（每个 agent）** | 向中转注册；本端发群消息后上报；收到广播后插入本机群聊记录。 |
| **Agent 本体** | 无；不感知中转、注册与插入。 |

---

## 3. 数据流

1. **注册 / 建长连接**：飞书插件启动时，携带统一 key 与中转建立长连接（如 WebSocket），并提交 `agent_id`。中转将连接与 agent_id 加入注册表。
2. **发言上报**：插件在飞书群发完消息后，经同一连接（或同机 HTTP）向中转发送：`agent_id`、`thread_id`（如 `feishu:group:oc_xxx`）、`content`、`ts`、可选 `message_id`（用于去重）。
3. **广播**：中转将本条发言通过长连接推送给**除发言人外**的其它已注册 agent。
4. **插入**：插件在长连接上收到 payload（`thread_id`、`sender_agent_id`、`content`、`ts`），在本机把这条作为「其它 agent 发言」追加到对应 thread 的 history，**不触发外脑回复**。

---

## 4. 鉴权（极简）

- **单一 key**：中转服务器上 hardcode 一个 key（或从环境变量读 `RELAY_KEY`）。Agent 飞书插件配置同一个 key。
- **用法**：注册/建连、上报时带上该 key（如 Header 或首包），中转校验 key 一致即接受；不区分注册 key / 广播 key，就一个 key。
- **配置**：中转侧 `RELAY_KEY`；插件侧 `RELAY_URL`、`RELAY_AGENT_ID`、`RELAY_KEY`（与中转一致）。

---

## 5. 边界与约定

- **只做「插入群聊记录」**：收到广播的插件只向 thread history 追加一条（role 等价于「其它用户/其它 agent」），不触发 LLM、不发起飞书发送。
- **去重**：同一发言可能因重试被广播多次，插件或中转可用 `message_id` / `(thread_id, sender, ts)` 做幂等。
- **thread_id 跨 agent 一致**：见下节。
- **Agent 无感**：注册、上报、接收与插入全部在飞书插件/适配层完成，不暴露给外脑或内脑逻辑。

---

## 6. thread_id 是否跨 agent 唯一（同一聊天是否一致）

**结论：是，同一聊天在不同 agent 上会得到相同的 thread_id。**

当前约定（与代码一致）：

- **格式**：`<channel>:<type>:<平台原生 ID>`，例如：
  - 群聊：`feishu:group:<chat_id>`
  - 私信：`feishu:dm:<open_id>`
- **来源**：群聊的 `chat_id`、私信的 `open_id` 均来自飞书平台，**同一群/同一会话在飞书侧只有一个 ID**。
- 因此：只要各 agent 的飞书插件都按上述规则用「飞书给的 chat_id / open_id」拼 thread_id，**同一飞书群在所有 agent 上 thread_id 相同**，广播时用这个 thread_id 插入，大家都能对到同一条会话。

无需各 agent 自己发明 ID；只要统一用平台原生 ID，即自然跨全局一致。

---

## 7. 鉴权与长连接

- **广播必须走长连接**：既然要实时广播，agent 与中转之间用长连接（如 WebSocket）接收推送；注册时提交的是「建立好的连接」或「由中转主动建连」的端点，而不是 HTTP callback URL。
- **鉴权极简**：中转服务器上 **hardcode 一个 key**（或从环境变量读一个固定 `RELAY_KEY`）。Agent 侧配置同一个 key；注册、建连、发上报时带上该 key，中转校验 key 一致即通过。不区分「注册用 key / 广播用 key」，就一个 key，双方一致即可。

配置示例：
- 中转侧：写死或配置 `RELAY_KEY=xxx`。
- Agent 飞书插件侧：配置 `RELAY_URL`、`RELAY_AGENT_ID`、`RELAY_KEY`（与中转一致）。注册/上报时在 Header 或首包带上 `RELAY_KEY`；建长连接时也可带同一 key 做鉴权。

---

## 8. 小结

- **一个中转服务器**：注册表 + 收到发言后经长连接向其它 agent 广播。
- **各 agent 仅在飞书插件侧**：用同一 key 注册、建长连接收广播；发群消息后上报；收到广播后插入本机群聊记录。
- **thread_id**：按 `<channel>:<type>:<平台原生 ID>` 生成，同一飞书群在各 agent 上一致。
- **鉴权**：中转 hardcode 一个 key，agent 配同一 key 即可。
