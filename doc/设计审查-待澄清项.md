# 设计审查：待澄清项（已按决策更新）

以下项已按最新决策在主设计文档与待细化清单中定稿；本文档保留原问题与结论，便于追溯。

---

## 1. Agent id 的派生方式 ✅ 已定

- **结论**：用**哈希**派生，实现自定（如 `SHA256(MAC + path)` 取前 16 字符）。主设计 §3、§6；待细化 §2.2。

---

## 2. 工作区目录 vs 记忆目录 ✅ 已定

- **结论**：
  - **临时目录**：Agent 的**配置、记忆、Soul** 均在**全局临时目录**下按 agent id 分子目录（`<全局临时目录>/<agent_id>/`）。
  - **工作目录**：与临时目录**分离**，表示 **agent 的操作对象**（另一项目的源码根目录）；read_file、write_file、shell_exec 等限定在工作目录或白名单内。
- 主设计 §3、§6；待细化 §2.2。

---

## 3. Soul 文件位置 ✅ 已定

- **结论**：Soul 文件在该 agent 的**临时目录**内（与配置、记忆同目录，如 `SOUL.md`）。主设计 §5、§6；待细化 §8.1。

---

## 4. 单次循环时 input 的来源与端点

- **结论**：端点机制已定「同时支持启动时指定与运行时注册」（主设计 §7；待细化 §1.3）。单次 run 读哪个端点、多端点时主端点约定仍可在实现时定（如默认主端点）。

---

## 5. 无 input 时本轮 user 输入 ✅ 已定

- **结论**：**无 input 时使用 SCL 控制提示词**（如 HEARTBEAT 检查清单或等效）作为本轮输入；SCL 本身是循环，可据此执行定时/巡检。主设计 §9、§9.1；待细化 §3.2。

---

## 6. Control 与能力缺口自举的时序 ✅ 已定

- **结论**：**下轮再自举**。本轮仅标记缺口；下一轮再执行 web_search、write_file 等自举流程，然后重试原任务。便于实现。主设计 §8.3；待细化 §5.1。

---

## 7. MEMORY.md 与 Mem0

- **结论**：记忆以**接口**形式对接，SCL 侧与本地实现通过接口交互，两边不关心具体实现；长期记忆由 Mem0 与本地实现按接口约定对接。主设计 §3（记忆与 SCL）；待细化 §6.1。

---

## 8. 错误时 output 的写法 ✅ 已定

- **结论**：**错误视为正常结果写入返回值**；不单独走异常通道，SCL 有能力处理错误。主设计 §12；待细化 §9.1。

---

## 9. 全局临时目录与身份路径

- **结论**：身份路径可指向该 agent 的临时目录（或用于唯一标识）；工作目录单独指定。实现时可约定 `--dir` 为临时目录路径或身份标识。主设计 §6。

---

## 10. 小结

| 项 | 状态 |
|----|------|
| Agent id 派生 | 已定：哈希，实现自定 |
| 临时目录 vs 工作目录 | 已定：临时目录=配置/记忆/Soul；工作目录=操作对象（另一项目） |
| Soul 位置 | 已定：临时目录内 |
| 无 input 输入 | 已定：SCL 控制提示词 |
| 端点机制 | 已定：启动时指定 + 运行时注册 |
| 能力缺口自举 | 已定：下轮再自举 |
| 错误与返回值 | 已定：正常结果写入 output，SCL 处理 |

其余细节见《轻量多 Agent 系统设计》与《待细化设计清单》。

## 仍可留待实现时约定的点（不阻塞开发）

以下在文档中未完全固定，实现时选一种即可，不影响主流程一致性。

| 点 | 说明 | 实现时可选 |
|----|------|------------|
| **身份路径与临时目录关系** | 身份由 (MAC, path) 确定，agent_id = hash(…)，临时目录 = 全局临时目录/<agent_id>/。path 由谁提供？ | `--dir` 传「身份路径」（任意字符串），系统用其算 agent_id 并使用固定规则 `全局临时目录/<agent_id>/` 作为临时目录；或 `--dir` 直接传临时目录路径，agent_id 从目录名/配置读。 |
| **工作目录如何传入** | 工作目录与临时目录分离，需单独指定。 | CLI `--workspace`、配置文件、或临时目录内 config 文件；子 agent 可由父在 input 中传入工作目录路径。 |
| **全局临时目录取值** | 根目录从哪来？ | 环境变量（如 `OPENKURONEKO_TMP`）、配置文件、或系统 temp 默认。 |
| **单次 run 的 input/output 端点** | 多端点时单次 run 读/写哪个？ | 默认主端点；或由启动参数/配置指定。 |
| **input/output 内容格式** | 纯文本 vs 结构化。 | 待细化 §1.2 已列；先做纯文本 + 读后消费 + 写覆盖即可。 |

以上在实现前或实现早期定一种并写在配置/CLI 文档即可，无需再改主设计共识。
