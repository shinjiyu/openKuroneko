# 实现前待细化设计清单

基于《轻量多 Agent 系统设计》文档，以下细节在动手实现前需要先做设计决策或约定，否则容易在 I/O 协议、生命周期、错误处理上出现歧义或返工。

---

## 1. Input / Output 接口协议

### 1.1 已定：接口抽象与实现可替换（实现约定）

- **接口定义**：input/output 是**接口**，不是绑定到真实文件。约定为「可读的 input、可写的 output」，语义为单生产者-单消费者。具体由**任务/运行时实现**：可以是真实文件、内存队列、消息队列、RPC channel 等，只要满足同一接口契约。
- **单生产者-单消费者**：每个 input 端点仅允许**一个**写者、每个 output 端点仅允许**一个**读者。不允许多写、多读。
- **多端点**：一个 agent 可配置多组 input/output 端点（命名示例：`in/user`、`out/user` 与 `in/agent_b`、`out/agent_b`）。每组端点仍为一写一读；通过增加端点来增加「通道」。端点的具体形态（路径、队列名、句柄等）由实现决定。
- **约定**：由配置或部署保证「谁是该端点的唯一生产者/消费者」；实现不做锁与多写兼容，违反约定即行为未定义。

### 1.2 待定

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **接口形态** | 抽象层暴露的 API（给 agent 与调用方）。 | 如 `readInput(endpointId?)` / `writeOutput(content, endpointId?)`；或流式 `getInputStream()` / `getOutputStream()`；由实现提供具体载体。 |
| **格式** | 读写内容为纯文本还是结构化？ | 纯文本（简单）；JSON/JSONL（可带元数据、多轮、来源标识）。 |
| **编码与换行** | 统一约定（当内容为文本时）。 | UTF-8；换行用 `\n`。 |
| **input 消费语义** | Loop 每轮读完后接口侧行为？ | 清空/ack/consume 一条（依实现：文件可 truncate，队列可 dequeue）。 |
| **output 写入语义** | 覆盖还是追加？调用方如何知道「有新结果」？ | 覆盖（简单）；或「覆盖 + 可选 metadata」供轮询。 |
| **多端点命名** | 多组 in/out 的标识与默认端点。 | 如 `in/<endpoint_id>`、`out/<endpoint_id>`；需约定「主端点」名称（如 `user` 或 `default`）供 reply_to_user 等写默认 output。 |

### 1.3 已定：端点来源机制

- 提供机制**同时支持**「启动时指定」（CLI、配置文件）与「运行时注册」（API、回调注册新端点）。

**建议**：在代码里先定义 input/output 的**接口类型**（或 trait），再提供至少一种实现（如文件实现）；协议格式可先定「最小版」（如纯文本 + 读后消费 + 写覆盖），在 doc 里写清。

## 2. Agent 身份与运行上下文

### 2.1 已定：身份与路径独占（实现约定）

- **身份唯一性**：Agent 身份由**本机 MAC 地址 + 路径**唯一确定。同一 (MAC, path) 即同一 agent；不同机器或不同路径即不同 agent。
- **路径独占**：**同一路径上不允许同时运行两个 agent**。一个路径在任意时刻至多一个运行中的 agent 实例；实现可用锁、pid 文件等保证互斥。
- **单次 run 的路径**：`--once --dir <path>` 的 `<path>` 即该次 run 的**身份路径**（当实现为文件时，也是 input/output、工作区所在目录）。

### 2.2 已定：临时目录 vs 工作目录、Soul 位置

- **临时目录**：Agent 的**配置、记忆、Soul** 均在**全局临时目录**下按 agent id 分子目录（如 `<全局临时目录>/<agent_id>/`）。Soul 文件也在该目录内（如 `SOUL.md`）。
- **工作目录**：与临时目录分离，表示 **agent 的操作对象**（另一项目的源码根目录）。read_file、write_file、shell_exec 等限定在工作目录或白名单内。
- **Agent id**：由身份（MAC + 路径）**哈希**派生，实现自定（如 SHA256 取前 16 字符）。

### 2.3 待定

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **非文件实现** | 非文件实现下「路径」的形态。 | 路径可为逻辑名，由运行时映射到具体 input/output 实现；MAC + 该逻辑路径仍唯一确定身份。 |

## 3. Loop 行为细节

### 3.1 已定：三种循环模式 + R-CCAM/ReCAP（实现约定）

参考 [Structured-Cognitive-Loop-Skills](https://github.com/shinjiyu/Structured-Cognitive-Loop-Skills)：

- **三种模式**：**1. 快速循环**（上一轮 LLM 结束立即下一轮，带防空转退避）；**2. 定时循环**（每隔 N 或 cron）；**3. 单次循环**（跑一轮退出）。
- **单次内步骤**：每轮 = R(Retrieval) → C(Cognition，含 ReCAP 分解/选节点) → C(Control 验证 + 能力缺口自举) → A(Action) → M(Memory 写回 + reply_to_user)。
- **快速循环防空转**：若连续 N 轮无新 input 且无实质性工具调用/状态变更，则退避（如指数增加间隔），避免空转。

### 3.2 待定

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **退避参数** | N 轮、退避上限、退避曲线。 | 如连续 3 轮空转则 2s→4s→8s，上限 60s。 |
| **定时周期** | 默认间隔与 cron 配置来源。 | 配置文件或 CLI 参数；如 `--interval 30m`。 |
| **并发与排队** | 上一轮未结束新 input 到达。 | 忽略本轮新 input，下轮再读；或排队依次执行。 |
| **空 input** | 定时/快速循环下无 input 是否跑。 | 已定：无 input 时本轮输入使用 SCL 控制提示词（如 HEARTBEAT 检查清单）；SCL 循环可跑日报/巡检。 |

## 5. 工具与 pi-mono 集成

### 5.1 已定：Work Agent 工具列表（实现约定）

需实现的工具（参考 SCL 原子能力与 Skill）：

- **read_file / write_file / edit_file**（或 read+write 替代 edit）、**shell_exec**、**web_search**、**get_time**：基础与感知。
- **reply_to_user(content)**：写主 output 端点。
- **run_agent**：exec 同程序 `--once --dir <path>`，写子 input、等退出、读子 output。
- **结构化状态**：TASKS（子任务树、状态、执行摘要）的读写约定，跨轮可靠。
- **Mem0**：add/search，以 agent_id 为 user_id 隔离。
- **capability_gap_handler**：元规则；**本轮仅标记缺口，下轮再执行自举**（web_search + write_file 等），便于实现。

### 5.2 待定（pi-mono 集成）

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **输入注入** | 每轮 run 时 input 如何交给 pi-mono？ | 作为本轮唯一 user message；或追加到会话末尾（Loop 复用会话时）。 |
| **工具挂载** | 哪些由 pi-mono 自带、哪些本系统封装？ | 若 pi-mono 有 read/write/shell 则直接用；reply_to_user、run_agent、Mem0 本系统实现并注入。 |
| **会话生命周期** | 单次 run 与 Loop 的会话是否复用？ | 单次 = 新会话；Loop = 可选复用 + 近期记忆摘要注入 system。 |

## 6. 第二层近期记忆

### 6.1 已定：存储位置与记忆接口（实现约定）

- **存储**：记忆（近期摘要、Daily Log、TASKS 等）在**全局临时目录**下按 **agent id** 分子目录，与配置、Soul 同处。
- **记忆与 SCL**：SCL 的 memory 语义按本项目记忆实现做调整；**记忆以接口形式对接**，SCL 与本地实现通过接口交互，两边不关心具体存储。
- Agent id 由身份（MAC + 路径）**哈希**派生（实现自定）。

### 6.2 待定

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **格式与容量** | 近期记忆文件格式、N 条或 T 天淘汰。 | 如 `recent_memory.jsonl`，最近 K 条；或 Daily Log 按日文件。 |
| **写入/注入时机** | 每轮结束写摘要、每轮开始读入 system。 | 与 R-CCAM 的 M/R 阶段对齐。 |

## 7. Mem0 使用约定

### 7.1 已定：全局单实例 + 按 agent 隔离（实现约定）

- **部署**：全局只启动**一个** Mem0 服务；各 agent 共用该服务。
- **隔离**：各 agent **自行隔离**：Mem0 调用时使用 **user_id = agent_id**（或等价命名空间），所有 add/search 带该 user_id，Mem0 按 user_id 逻辑隔离，agent 间数据互不串。
- **配置**：Mem0 endpoint、API key 等由全局配置或环境变量提供；agent 仅传入自身 agent_id 作为 user_id。

### 7.2 待定

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **add/search 时机** | 每轮自动 vs 按需调用。 | 每轮结束自动 add 摘要；每轮开始自动 search 注入 system；或部分按需。 |

## 8. Soul 格式与热载边界

### 8.1 已定

- **位置**：Soul 文件在该 agent 的**临时目录**内（与配置、记忆同目录，如 `SOUL.md`）。每 agent 一份。

### 8.2 待定

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **文件格式** | 纯文本、YAML、JSON？ | 纯文本（一段 system 描述）；YAML（可拆为人设、规则、默认行为多字段）。 |
| **热载生效时机** | 改 soul 后何时生效？ | 仅**下一轮** run 的 system 使用新 soul；当前轮不中断（安全）。 |
| **与 system 的拼接** | soul + 记忆摘要 + 其他如何组合？ | 约定顺序：soul 全文 + 近期记忆摘要 + Mem0 检索结果 + 固定尾段。 |

**建议**：soul 用纯文本或 YAML，热载仅影响下一轮；在 doc 里写清 system 的拼接顺序。

## 9. 错误与边界行为

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **run 中崩溃** | output 未写或写一半，调用方如何知道？ | 单次 run：父进程看 exit code；Loop：本轮不写 output，下一轮可读 input 重试；或写 `output.failed` + 简短错误信息。 |
| **input 非法** | 格式错误、空、或超长？ | 忽略并写 output 提示「input 无效」；或写日志不写 output，由调用方超时判断。 |
| **LLM/API 不可用** | 重试、退避、降级？ | 重试 1～2 次后写 output 报错；或写 status 文件「running / failed」供 UI 轮询。 |
| **子 agent 超时** | exec 子 agent 时父 agent 是否卡住？ | 父进程 exec 时设 timeout；超时则 kill 子进程，父 agent 读到的 output 可能不完整，需在 output 或 status 中标记「timeout」。 |

### 9.1 已定

- **错误视为正常结果**：错误作为**正常结果**写入 output（返回值），不单独走异常通道；SCL 有能力在循环内处理错误并决定下一步。

**建议**：成功与失败均通过同一 output 出口返回；调用方/SCL 根据内容判断并处理。

## 10. 安全与权限（可选但建议先约定）

## 11. 可观测与调试

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **run 日志** | 是否每轮落盘？ | 每轮在 agent 目录写 `logs/run_<ts>.log` 或 append 到 `agent.log`。 |
| **链路标识** | 用户请求 → 某 work run → 子 agent run 如何串联？ | 可选：input 中带 `request_id`；output 和子 agent 的 input 透传该 id；日志中打印 request_id。 |

**建议**：至少保留「每轮 run 一条日志 + 可选 request_id 透传」，便于排查多 agent 串联问题。

## 12. 配置与部署

| 待定项 | 说明 | 可选方案示例 |
|--------|------|-----------------------------|
| **LLM 配置** | 模型、API key、endpoint 从哪读？ | 环境变量；或 `config.yaml` / `.env`；聊天与 work 可用不同 config。 |
| **多实例** | 多个 work agent 如何起？ | 同一二进制，不同 `--dir`（及可选 `--id`）；或同一 binary，多个 systemd/进程，各读各的 config/soul。 |

**建议**：统一约定「一个 agent 一个目录，目录内可含 config 或继承全局 config」；在 doc 里写清配置优先级（如 dir/config > 全局 > 环境变量）。
